## Наблюдения (некритично, на будущее)

1. **SECRET_KEY=None при незаданной переменной.** `create_app()` берёт
   `os.environ.get("TODO_SECRET_KEY")`; если переменная не задана, значение
   будет `None`, и первый вызов `flash` упадёт с `RuntimeError` на запросе.
   Правило architecture.md соблюдено (ключ не в коде). Рекомендация: при
   старте логировать warning, если ключ не задан.
2. **Переиспользование id после удаления.** `_next_id` = `max(id) + 1`.
   После удаления последней задачи новый id может совпасть с прежним.
   Уникальность среди текущих задач сохраняется — для одного пользователя
   приемлемо.
3. **Нет защиты от конкурентного доступа.** Каждая мутация выполняет
   load → mutate → save без блокировки файла. Спецификация оговаривает
   одного локального пользователя — приемлемо, отмечено как известное
   ограничение.
4. **load_tasks не валидирует структуру задач.** Проверяется только то, что
   JSON — список; поля `id/text/done` у элементов не проверяются. Валидный,
   но некорректный по структуре файл пройдёт загрузку и сломает service.
5. **Нет обратной связи при операции с несуществующим id.** `toggle`/`delete`
   для отсутствующего id возвращают `False`, маршрут молча делает redirect.
   Минорный UX-момент.
6. **В репозитории нет `.gitignore`.** Генерируемые `tasks.json`,
   `__pycache__/`, `.pytest_cache/` могут случайно попасть в коммиты.
   Рекомендация для harness-гигиены.

Наблюдения 1, 4, 6 стоит занести в `docs/plans/tech_debt.md` при следующем
проходе Пайплайна 06 (cleanup).
