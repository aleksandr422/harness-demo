# HARNESS SUMMARY — закрытие E2E-цикла

Итоговый отчёт по демо агентского harness-пайплайна. Демо построено по
мотивам статьи OpenAI «Harness Engineering»: показать, как агент проходит
полный продуктовый цикл внутри инженерно выстроенного «harness».

- **Коммитов:** 20 (атомарные, с осмысленными сообщениями).
- **Тестов:** 24 pytest-теста, все зелёные; покрыта каждая публичная
  функция модулей `src/`.
- **Кода продукта:** 5 файлов в `src/` (4 Python-модуля + HTML-шаблон),
  каждый ≤ 200 строк.
- **Тест-файлов:** 4 (`conftest.py` + 3 модуля тестов).
- **Релиз:** v0.1.0.

## Что за продукт получился

Минимальное **To-Do приложение** на Python + Flask:

- добавление / отметка выполнения / удаление задач;
- хранение в JSON-файле на диске (атомарная запись, без БД);
- одностраничный веб-интерфейс, без авторизации;
- валидация ввода (непустой текст, ≤ 200 символов);
- архитектура слоями: `storage` → `service` → `routes` → `app`.

Проверено end-to-end: приложение поднимается, все операции и сценарии
валидации работают (Пайплайн 05).

## Соответствие подходу «Harness Engineering»

Идея harness engineering — что качество агентской работы определяется не
только моделью, но и «harness» вокруг неё: контекстом, инструментами,
обратными связями и средой. Что из этого видно в демо:

- **Harness как первоклассный артефакт.** `AGENTS.md`, `docs/rules/`,
  пайплайны — это явно описанная среда, которую агент обязан прочитать
  перед работой. Контекст не в голове агента, а в репозитории.
- **Стадии с верификационными гейтами.** Каждый пайплайн имеет чёткий
  вход/выход и проверку: линтер после кода, pytest в ревью, HTTP-проверки
  при деплое. Агент не идёт дальше, пока гейт не зелёный.
- **Замкнутые петли обратной связи.** Линтер и тесты — автоматические
  верификаторы; агент прогоняет их и сам исправляется по результату.
- **Декомпозиция.** ExecPlan дробит работу на атомарные задачи (≤ 1 файл),
  упорядоченные по зависимостям — снижает риск и упрощает проверку.
- **Эволюция самого harness.** Агент улучшал инструментарий (линтер,
  `.gitignore`, доки) — harness не статичен, он чинится по ходу дела.
- **Durable state.** Артефакты в `docs/` переживают перезапуск окружения:
  harness переносит контекст между сессиями, а не теряет его.

Демо закрывает полный цикл: идея → спецификация → план → код → ревью →
релиз → очистка — внутри воспроизводимого, самоулучшающегося harness.
